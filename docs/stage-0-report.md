# 阶段 0 技术验证报告

验证上游：`libnoname/noname`，固定提交 `632d2d3c8da2893466a8c440a18861c9ed49813d`（2026-07-13）。

运行审计：

```powershell
node scripts/stage0-audit.mjs
```

## 实验 1：浏览器客户端能否由外部输入驱动

结论：**可以，但上游采用房主浏览器权威模型，不能原样用于本项目。**

证据：上游 `apps/core/noname/game/index.js` 将 WebSocket 的 `onmessage` 绑定到游戏消息处理器，并通过 `game.send` 发送选择结果；`packages/server/src/server/createServer.ts` 的 `send/onmessage` 只在房主与从属客户端间转发消息。换言之，远端输入能驱动游戏，但规则仍在房主浏览器执行。

本项目处理：浏览器只发送 `useCard/respond/endTurn` 命令；服务端校验并生成每名玩家各自的裁剪视图。

## 实验 2：无名杀逻辑能否在无 DOM Node.js 运行

结论：**不能直接运行。**

静态扫描发现 43 个核心源文件引用 `window/document/localStorage/HTMLElement`。直接导入开发入口还依赖其浏览器构建别名和生成产物；即使补齐别名，初始化、UI、玩家对象和事件循环仍与 DOM 相互引用。

本项目处理：已抽离 `@sgs/headless-engine`。该包不引用浏览器 API，具备命令分发、私有视图、快照恢复和扩展技能触发接口。

## 实验 3：相同输入和随机种子能否复现

结论：**上游原样不能保证。**

核心目录中有 14 个文件直接使用 `Math.random()`，随机状态不可序列化，也没有统一种子入口。

本项目处理：所有牌堆、身份和选将随机均经过可序列化的 xorshift32。自动测试验证：

- 同一种子与命令序列生成逐字节相同快照。
- 从快照恢复后继续执行生成相同快照。
- 回放按命令步数从初始种子重新执行，而非播放伪造日志。

## 架构决策

不使用上游房主权威联机模式，也不在 Node 中伪造完整 DOM。无名杀源码作为 GPLv3 内容/API 兼容参考，规则执行迁移到可测试的服务端模块；后续兼容工作在 `noname-adapter` 中完成。

## 后续验证：兼容技能模块切片

完整浏览器入口不能直接在 Node 运行，并不代表所有上游规则代码都必须重写。进一步审计发现，武将 `skill.js` 顶层主要声明技能对象，DOM 与 `ui` 调用多数位于尚未执行的技能函数内部。

因此 `@sgs/noname-adapter` 增加了固定上游技能模块加载器：在不初始化浏览器的情况下加载真实技能声明，通过宿主代理执行技能函数，并把其中的 `Math.random` 替换为可快照种子随机源。详细边界和后续工作见 [无名杀兼容运行时](noname-compatibility-runtime.md)。
